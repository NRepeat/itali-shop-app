## Three open bugs to fix (none started yet):

### Bug 1: Order #6340 cancelled but status not updated on frontend
- **Root cause identified**: After `orderCancel` Shopify mutation, `displayFulfillmentStatus` stays `"UNFULFILLED"` — Shopify doesn't set it to `"CANCELLED"`. The proxy `prod.nnninc.uk/api/order/:id` (which is itali-shop-app's `/api/order/:id` route) returns `fulfillmentStatus` from Shopify which is `UNFULFILLED`, not cancelled.
- **Fix needed**: Find `/api/order/:id` route in itali-shop-app, check if it reads `cancelledAt` field from Shopify order. If not, add logic: if `cancelledAt` is set → return `fulfillmentStatus: "СКАСОВАНО"`.
- **Frontend**: `OrderStatusBadge` already handles `"СКАСОВАНО"` → red badge ✓

### Bug 2: keyCRM webhook must always return 200
- **Root cause**: If any error occurs in `handleKeyCrmOrderStatusChange`, the webhook returns 500, keyCRM retries and eventually times out.
- **Fix needed**: `app/routes/api.keycrm-webhook.ts` — wrap entire handler body in try/catch, always return 200. Log errors but don't propagate them.

### Bug 3: Tracking number in keyCRM webhook is in `context.shipping.tracking_code`, not `context.ttn`
- **Info from user**: keyCRM webhook payload structure for order status change:
  ```json
  { "shipping": { "tracking_code": "20450000000000", "tracking_status": "Shipping" } }
  ```
  The `context.ttn` path we use in `keycrm-shopify-sync.service.ts` is WRONG.
- **Fix needed**: In `handleKeyCrmOrderStatusChange`, change `context?.ttn` → `context?.shipping?.tracking_code`
- Also need to pass trackingNumber to Shopify order fulfillment/tracking update (currently only passed to eSputnik queue)
